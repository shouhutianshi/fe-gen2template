你需要站在更高维度重新审视当前任务。不要机械地把 PRD/系分/测分拆成子任务——那只做了翻译，没有做设计。

## 核心要求

### 1. 理解真实目标

- 用户给到的文档很多地方是点到为止的，没有展开的部分才是你需要补全的
- 先用一句话说清楚"这个功能最终要解决什么问题"，确认理解没有跑偏
- 如果理解不了目标，先问清楚，不要急着拆任务

### 2. 补全交互设计

- 把用户在系统里的行为路径想清楚：每一步怎么触发、状态怎么变化、用户看到什么反馈
- 覆盖正常路径和异常路径（出错时用户看到什么、怎么恢复）
- 细到可以直接指导前端实现的程度

### 3. 补全接口与数据设计

- 补出交互路径对应的接口列表：每个接口的入参、出参、错误码
- 补出数据流转：接口之间怎么串联，数据怎么传递
- 补出数据存储设计：需要哪些表/字段，状态怎么流转

### 4. 落地验证

- 从"能不能被完整实现"的角度做一轮 review
- 检查有没有遗漏的边界条件、未定义的状态、断掉的数据链路
- 如果发现文档中的设计有矛盾或不可行之处，直接指出并给出替代方案

## 输出格式

按以下结构输出，不要省略任何一节：

```
## 目标
[一句话说清楚这个功能要解决什么问题]

## 用户行为路径
[完整的交互流程，包含每一步的状态和反馈]

## 接口设计
[每个接口的方法、路径、入参、出参、错误码]

## 数据结构
[表结构/字段设计，状态枚举，关键索引]

## 落地检查
[边界条件清单、风险点、需要确认的假设]
```

如果当前没有具体的 PRD/系分/测分上下文，先问用户要。

并行调查用户描述的问题或需求。核心原则：**同一时间派出多个 Agent 各自查不同方向，然后汇总结论**，而不是串行猜测。

## 流程

### 1. 拆解调查方向

根据用户描述的问题，拆出 2-4 个独立的调查方向。每个方向一个 Agent，互不依赖。

常见拆分策略：

| 场景 | Agent 1 | Agent 2 | Agent 3 |
|---|---|---|---|
| Bug 排查 | 查网络请求/API 层 | 查状态管理/数据流 | 查组件逻辑/UI 渲染 |
| 性能问题 | 查大数据源/慢查询 | 查重复渲染/计算 | 查资源加载/构建体积 |
| 重构摸底 | 查使用范围（谁调用了） | 查测试覆盖 | 查配置和依赖关系 |
| 新需求评估 | 读 PRD 提取需求点 | 读现有代码看影响面 | 查接口文档/YApi |
| 集成问题 | 查前端代码 | 查后端接口响应 | 查环境配置/代理 |
| 构建失败 | 查错误日志定位文件 | 查依赖版本兼容性 | 查配置文件正确性 |

### 2. 并行派发

在**同一个工具调用块**中同时启动所有 Agent，不要等一个完成再启动下一个。每个 Agent 的 prompt 要包含：

- 明确的调查范围（只查什么）
- 需要读取的文件路径（如果已知）
- 输出格式要求：发现的事实、相关代码位置、结论

```
使用 Agent 工具，subagent_type 为 "Explore"，同时启动 N 个调查 Agent。
所有 Agent 调用放在同一个工具调用块中并行执行。
```

### 3. 汇总分析

所有 Agent 返回后：

1. 列出每个方向的发现
2. 交叉验证：不同 Agent 的发现是否一致
3. 标注矛盾点：信息冲突的列出来让用户判断
4. 给出综合结论和建议的下一步操作

### 4. 输出格式

```markdown
## 调查结论

### 方向 1：xxx
- 发现：...
- 相关代码：`file:line`
- 结论：...

### 方向 2：xxx
- 发现：...
- 相关代码：`file:line`
- 结论：...

### 综合
- 根因/核心发现：...
- 建议：...
- 矛盾点（如有）：...
```

## 注意事项

- 调查 Agent 只读不写（subagent_type: "Explore"），不做任何修改
- 如果方向之间有依赖（比如 B 需要 A 的结论），不要强行并行，先跑 A 再跑 B
- 单个 Agent 超过 30 秒没返回结果，不要等待，先处理已返回的结论
- 调查完成后，如果需要修复，切换到正常开发模式（不用并行 Agent）

# 团队审查

你是**协调器**。创建一个代理团队并分发审查员、验证员和修复员代理。这对多轮迭代至关重要：代理是一次性的，因此你的上下文只会通过结构化问题列表增长，而非原始文件内容——避免因上下文压缩而丢失对前几轮的追踪。切勿直接修改文件。仅为了仲裁和诊断而读取代码。

审查循环专为**不间断的多轮迭代**而设计。在让用户参与之前，始终处理所有可自动修复的问题。在确认（第 6 阶段）或报告（第 7 阶段）之前，切勿暂停询问用户任何事情。

审查员-验证员对抗对是核心质量机制：审查员发现问题，验证员质疑它们。这种双向检查显著降低了误报率。审查员和验证员绝不能看到彼此的输出或共享对话历史。

## 来自 SKILL.md 的输入

- `FIX_MODE`: low | low_medium | full

## 参考资料

| 文件                 | 用途                               |
| -------------------- | ---------------------------------- |
| `code-checklist.md`  | 代码审查清单                       |
| `doc-checklist.md`   | 文档审查清单                       |
| `judgment-matrix.md` | 风险等级、值得修复的标准、特殊规则 |

## 流程

```
范围
  │
  ↓
┌─────────── 审查循环 ───────────┐
│                                   │
│  审查 → 过滤 → 修复/验证   │
│    ↑                     │        │
│    └── 新问题 ───────
│                                   │
└───────────────────────────────────┘
  │ 无新问题
  ↓
pending/failed? ──no──→ 报告
  │ yes
  ↓
确认 ──all skipped──→ 报告
  │ approved
  ↓
修复/验证 ──→ 审查循环 ↑
```

只要发现新问题，审查循环就会重复。每一轮都是全新的完整审查——而非对先前修复的有针对性重新检查。

**确认后**：已批准的修复会经过修复/验证，然后**始终重新进入审查循环**（第 2 阶段），以捕获修复可能引入的任何问题。只有当一轮没有产生新问题时，循环才会退出到确认（如果仍有 pending/failed）或报告（如果没有任何剩余）。

---

## 第 1 阶段：范围

根据 `$ARGUMENTS` 确定要审查的差异：

- **空参数**：通过按顺序检查常见基础分支来找到基础分支：`main`、`master`。使用第一个存在的分支。获取分支差异：
  ```
  git merge-base origin/{base_branch} HEAD
  git diff <merge-base-sha>
  ```
- **提交哈希**（例如 `abc123`）：使用 `git rev-parse --verify` 验证，然后使用 `git show`。
- **提交范围**（例如 `abc123..def456` 或 `abc123...def456`）：验证两个端点。获取包含两个端点的差异：
  ```
  git diff A~1..B
  ```
- **文件/目录路径**：验证所有路径在磁盘上存在，然后读取文件内容。

如果差异为空 → 显示使用示例并退出：
`/cr`（未提交的更改或当前分支），
`/cr a1b2c3d`、`/cr a1b2c3d..e4f5g6h`、
`/cr src/foo.cpp`、`/cr 123`、`/cr https://github.com/.../pull/123`。

### 关联的 PR 评论（可选，尽力而为）

如果 `gh` 可用，检查当前分支是否有开放的 PR：

```
gh pr view --json number,state --jq 'select(.state == "OPEN") | .number' 2>/dev/null
```

如果存在开放的 PR，获取其行级审查评论：

```
gh api repos/{owner}/{repo}/pulls/{number}/comments
```

存储为 `PR_COMMENTS` 以便在审查步骤中进行验证。

### 构建基线

如果是仅文档则跳过。如果无法确定构建/测试命令，警告将跳过修复验证。否则运行构建 + 测试。失败 → 中止。

读取自基础分支以来的 git 日志以获取最近修复的上下文（避免重新标记先前 `/cr` 会话已修复的问题）。

### 模块划分

将范围内的文件划分为**审查模块**以进行并行审查。每个模块都是一个独立的逻辑单元。按部分/函数组拆分大文件；将相关的小文件分组在一起。将每个模块分类为 `code`、`doc` 或 `mixed`。

### 持久化状态

初始化 `CR_STATE_FILE`：如果 `.cr-cache/` 不存在则创建。从审查范围派生文件名：`.cr-cache/{branch}.md`（将 `/` 规范化为 `-`，例如 `feature/dom_text_box` → `feature-dom_text_box.md`）。

每次 `/cr` 调用都是一个**全新会话**。切勿从现有文件读取或恢复——每个会话都从零开始，进行新的完整审查。如果派生的文件名已存在（来自先前或并发会话的残留），找到最小的未使用数字后缀（`-2`、`-3`、...）并使用它。将所选路径记录为 `CR_STATE_FILE`。

写入包含会话信息（模式、阈值、文件列表、模块分配、更改的行范围、构建/测试命令）和问题部分（增量更新）的 CR_STATE_FILE。CR_STATE_FILE 由协调器拥有——团队代理永不读取或写入它。将任何未解决的问题记录到 CR_STATE_FILE 供用户审查。

**CR_STATE_FILE 中的问题格式**：

```markdown
# 问题

## 1. [简要描述]

- **状态**：pending | approved | fixed | failed | skipped
- **原因**：[为何未自动修复，例如"高于自动修复阈值（高风险）"、"修复失败：重试 2 次后仍出现构建错误"、"已回滚：引入了回归"]
- **风险**：low | medium | high
- **文件**：[文件路径：行号]
- **当前**：[代码现在的行为]
- **建议**：[修复将更改的内容——对于中等/高风险，包括协调器选择的特定方法及其原因]
- **影响**：[还有什么会受到影响]
```

状态值：

- `pending` — 已记录，等待在第 6 阶段由用户决定
- `approved` — 用户在第 6 阶段批准修复，发送到第 4 阶段
- `fixed` — 修复已应用并通过验证
- `failed` — 修复尝试但在重试后验证失败
- `skipped` — 用户拒绝或问题被拒绝（不重新报告）

## 第 2 阶段：审查

### 轮次不变量

- **恢复状态**：协调器在每轮开始时读取 CR_STATE_FILE。审查员不读取此文件。
- **已知问题排除列表**（第 2 轮+）：协调器从 CR_STATE_FILE 提取——每个条目仅包含文件路径、行范围和单行描述。无风险级别、无验证员裁决、无修复结果。
- **跨轮去重**（协调器，第 3 阶段）：以 CR_STATE_FILE 作为安全网进行去重。先前确认中被用户拒绝的问题也被排除。先前修复的问题**不被排除**——修复代码中的新问题是有效的新问题。

### 团队设置

为这一轮创建一个新团队。每轮都有一个全新的团队——不要重用先前轮次的代理（它们在团队关闭后会丢失上下文）。

- 每个模块一个审查员代理（`reviewer-N`）。
- 一个**验证员**代理（`verifier`），在所有模块间共享。

### 审查员提示

立场：**彻底**——尽可能发现真实问题，提交前自我验证。

每个审查员接收：

- **范围**：其模块的文件列表 + 更改的行范围。审查员根据需要自行获取差异和读取额外的上下文——协调器不传递原始差异或文件内容。
- **清单**：代码用 `code-checklist.md`，文档用 `doc-checklist.md`，混合的两者都用。
- **证据要求**：每个问题都必须有来自当前树的代码引用（文件:行 + 片段）。
- **已知问题排除**（第 2 轮+）：跳过与协调器排除列表匹配的问题。专注于发现新问题。
- **清单排除**：查看相应清单中的排除部分。上下文中加载的项目规则优先。
- **自检**：提交前，重新阅读相关代码并验证每个问题。标记为已确认或已撤回。仅提交已确认的问题。如果引用的路径/行不再存在，通过 `git diff --name-only` 或文件搜索定位正确的文件/路径，然后再报告。
- **输出格式**：`[文件:行] [A/B/C] — [描述] — [关键行]`

**PR 评论审查员**（当 `PR_COMMENTS` 存在时，仅第 1 轮）：一个额外的代理，用于根据当前代码验证 PR 审查评论。相同的输出格式，相同的验证流程。在后续轮次中跳过。

### 验证（流程）

立场：**对抗**——默认质疑审查员，主动寻找每个问题错误的原因。使用真实证据拒绝，如果成立则确认。此步骤是强制性的——协调器绝不能跳过它或自己执行验证。

验证员作为**流程**运行——它不会等待所有审查员完成。当每个审查员通过 SendMessage 发送报告时，协调器必须立即将其转发给验证员。转发规则：

- **逐字引用**：将审查员的原始 SendMessage 内容包裹在引用块中并按原样发送。
- **不重写**：不要总结、重新组织、将多个审查员报告合并到一条消息中，或添加协调器评论。
- **每个审查员一次转发**：每个审查员报告都是发送给验证员的单独消息。

在每个验证员提示中逐字包含以下内容：

```
你是一个代码审查验证员。你的立场是对抗性的——默认质疑审查员的结论，主动寻找问题可能错误的原因。你的工作是对每个问题进行压力测试，以便只有真实问题能够留存。

对于你收到的每个问题：

1. 阅读引用的代码（文件:行）和足够的周围上下文。
2. 主动尝试反驳问题：审查员的推理是否有缺陷？是否存在使这成为非问题的上下文（例如，调用者保证的不变量、平台约束、有意设计）？代码实际上是否像审查员声称的那样行事？寻找你能找到的最有力的反驳论点。
3. 每个问题的输出：
   - 裁决：REJECT 或 CONFIRM
   - 推理：对于 REJECT，陈述具体的反驳论点。对于 CONFIRM，简要说明你检查了什么以及为什么不存在有效的反驳论点。

重要约束：
- 你的反驳论点必须基于代码中的真实证据。不要捏造假设的辩护或发明在代码库中不可见的调用者保证。
- CONFIRM 裁决不是失败——这意味着审查员发现了一个真实问题，而你的挑战验证了它。
```

### 审查后

保持所有审查员活动（在第 4 阶段重用为修复员），关闭验证员 → 第 3 阶段。

---

## 第 3 阶段：过滤——仅协调器

进入此阶段之前，确认：（1）所有审查员都已提交最终报告；（2）验证员已对每个转发的发现给出 CONFIRM/REJECT 裁决。在两个条件都满足之前，不要继续。

你此处的立场是**中立**——不信任任何一方。将审查员报告和验证员反驳视为同等加权的输入。利用你的项目范围视角考虑跨模块影响、约定和局部审查员可能错过的架构意图。

### 3.1 去重

- 移除已在 CR_STATE_FILE 中的问题
- 移除跨审查员的重复（相同位置、相同主题）
- 移除先前确认中用户拒绝的问题

先前修复的问题不被排除——修复代码中的新问题是有效的。

### 3.2 存在检查

| 验证员裁决 | 操作                                                                 |
| ---------- | -------------------------------------------------------------------- |
| CONFIRM    | 合理性检查——验证描述是否与引用的代码匹配。如果看起来不对，读取代码。 |
| REJECT     | 读取代码。评估两个论点。仅当反驳论点合理时才删除。                   |

### 3.3 风险级别

查阅 `judgment-matrix.md` 了解风险级别评估、值得修复的标准、按风险级别的处理和特殊规则。

**修复方法**（仅中等/高风险）：指定所选的方法及其原因。记录在问题的 `Proposed` 字段中。低风险：单一明显修复，无指导。

### 3.4 路由——记录到 CR_STATE_FILE

所有已确认的问题都记录有风险级别。

| 风险 vs `FIX_MODE` | →                            |
| ------------------ | ---------------------------- |
| 等于或低于阈值     | 自动修复队列                 |
| 高于阈值           | CR_STATE_FILE 作为 `pending` |

- 跨模块影响：如果修复需要修复员模块之外的更新，创建后续修复任务。
- 先前已回滚的问题：本轮不再尝试。

如果自动修复队列非空则进入第 4 阶段；如果为空则进入第 5 阶段。
始终先自动修复符合条件的问题——在所有可自动修复的问题都已处理和验证之前，不要向用户展示 `pending` 问题。

---

## 第 4 阶段：修复/验证

此阶段是一个原子单元，由审查循环和确认后流程重用。

### 修复

立场：**精确**——完整正确地应用每个修复，绝不扩大范围。协调器绝不能直接应用修复。

**代理分配**：重用幸存的审查员作为修复员——每个审查员已经拥有其审查文件的上下文。协调器动态分配修复任务。协调器必须通过明确的文件列表进行分配，并确保一个文件在同一时间仅由一个修复员拥有：

- 审查员已读取的文件中的问题 → 分配给该审查员。
- 跨文件问题或没有匹配审查员的问题 → 分配给 `fixer-cross` 代理（如果需要则创建一个）。
- 跨模块修复：分配给拥有目标文件的审查员或 `fixer-cross`，并在任务中包含完整文件列表。
- 多文件重命名 → 分配给一个代理的单个原子任务。

一个代理可能接收多个修复任务，如果它覆盖多个文件。避免将同一文件分配给多个代理以防止并发编辑冲突。

每个修复员接收：

- 问题描述 + 文件路径 + 行范围（修复员自行读取文件）
- 来自 `Proposed` 字段的修复方法（中等/高风险）
- 修复员规则（在每个修复员提示中逐字包含）：

```
修复规则：
1. 修复每个问题后，立即：git commit --only <files> -m "message"
2. 仅修改协调器明确分配的文件。切勿使用 git add .
3. 如果修复需要更改未分配的文件，停止并向协调员报告以重新分配。
4. 提交消息：英文，120 字符以内，以句号结尾。
5. 如有疑问，跳过修复而不是冒险进行错误的更改。
6. 不要运行构建或测试。
7. 不要修改公共 API 函数签名或类定义（注释可以），除非协调器的问题描述明确要求 API 签名修复。
8. 每次修复后，检查更改是否影响分配文件内的相关注释或文档（函数/类文档注释、描述更改逻辑的内联注释）。如果是，在与修复相同的提交中更新它们。跨模块文档更新（README、spec 文件、其他模块）由协调器单独处理。
9. 完成后，报告每个修复的提交哈希并列出任何跳过的问题及跳过原因。
```

每个修复员按问题提交（每个修复一个提交）。

### 验证

等待所有修复员。运行构建 + 测试。

- 如果没有可用的构建/测试命令或仅文档模块则跳过。
- **通过** → 在 CR_STATE_FILE 中标记问题为 `fixed` → 第 5 阶段。
- **失败** → 使用 bisect 找到失败的提交，将其回滚，在责怪其他之前重新验证其余（一个错误的提交可能导致级联失败）。对于每个失败的问题：通过原始修复员代理重试，附上失败详细信息（最多重试 2 次），或回滚并标记为 `failed`。

---

## 第 5 阶段：继续？

### 步骤 1：关闭当前轮的团队

关闭团队以释放审查员和修复员。强制终止任何无响应的代理。

### 步骤 2：路由

**从上到下评估条件；采用第一个匹配项。**

| 条件                                     | →                       |
| ---------------------------------------- | ----------------------- |
| 本轮有任何问题被确认和修复               | 第 2 阶段（新的审查轮） |
| CR_STATE_FILE 中有 `pending` 或 `failed` | 第 6 阶段               |
| 否则                                     | 第 7 阶段               |

第一个条件意味着：如果本轮审查发现了真实问题并应用了修复，始终进行全新的完整审查以捕获前一轮可能遗漏的问题。这不是对修复的有针对性重新检查——这是对整个范围的全新团队进行的完整新审查。

---

## 第 6 阶段：确认

按风险分组展示 `pending` + `failed` 问题（高 → 低），每组内按文件路径排序：
`[数字] [文件:行] [风险] [原因] — [描述]`

然后要求用户使用**单一多选问题**选择要修复的问题，其中每个选项的标签是问题摘要（例如 `[风险] 文件:行 — 描述`）。用户在一个提示中检查多个选项。已选中 → `approved`，未选中 → `skipped`。

- **全部跳过** → 第 7 阶段。
- **任何已批准** → 第 4 阶段（修复/验证）。验证后，进入第 5 阶段（继续？）。如果发现新问题，重新进入完整审查循环（第 2 阶段）。如果没有新问题但仍有更多 `pending`/`failed`，返回此处（第 6 阶段）。如果没有任何剩余，进入第 7 阶段。

---

## 第 7 阶段：报告

强制终止任何仍在运行的代理。关闭团队。删除 CR_STATE_FILE。

摘要：

- 轮次和每轮统计
- 发现的问题 / 已修复 / 已跳过 / 失败
- 已回滚的问题及原因
- 最终测试结果
- 来自 PR 评论的问题（当 `PR_COMMENTS` 存在时）

# 文档审查清单

审查者需检查本清单中的所有项目。A/B/C 级别表示审查优先级 —
从 A 级（最高影响）开始，然后 B 级，最后 C 级。审查者提示会指定需要检查的级别。

**项目特定规则**：已在代理上下文中加载的项目规则优先于此通用清单。

---

## 优先级 A：准确性

文档包含错误、矛盾或不完整信息的问题。最高影响 — 优先审查这些项目。

**A1. 代码与文档一致性**

- 描述的行为是否与实际代码实现相符？
- 参数名称、类型和默认值是否与代码一致？
- 返回值和错误条件是否准确记录？
- 描述的算法或处理步骤是否与实现一致？

**A2. 事实正确性**

- 版本号、格式标识符和常量是否正确？
- 取值范围和约束是否准确？
- 枚举值及其含义是否与代码定义一致？
- 文件格式结构（字节偏移、字段大小等）是否正确？

**A3. 内部一致性**

- 描述同一概念的不同部分是否相互一致？
- 约束和规则在整篇文档中是否一致（例如，某处说"必须 >= 0"，而另一处暗示默认值为负数）？
- 当同一条规则出现在多个位置时，所有实例的含义是否完全相同？

**A4. 完整性**

- 是否记录了所有公共 API / 功能？
- 新增的功能或参数是否在文档中体现？
- 已移除或弃用的功能是否相应标记？
- 边缘情况和限制是否已记录？

**A5. 逻辑完整性**

- 是否穷尽地覆盖了所有条件分支（例如，"如果 A... 如果 B..." — 其他情况呢）？
- 顺序步骤是否完整，没有遗漏中间步骤？
- 是否识别了未定义行为（没有记录结果的输入组合）？

**A6. 引用有效性**

- 内部交叉引用是否指向现有的章节或文件？
- 外部链接 / URL 格式是否正确且未明显过时（例如，指向已知已失效的域名）？不要尝试验证可访问性。
- 引用的文件路径、工具名称和命令示例是否实际存在？

---

## 优先级 B：清晰度与结构

提高可读性、明确性和组织性的改进。中等影响。

**B1. 歧义检测**

- 是否存在可以有多种解释的描述？
- 条件陈述是否精确（"应该"与"必须"，"可能"与"将"）？
- 边界条件是否明确陈述（包含与不包含范围，"至少"与"恰好"）？

**B2. 简化**

- 能否在不失真意的情况下使冗长的描述更简洁？
- 是否存在重复相同信息的冗余段落或章节？
- 能否用表格、列表或示例替换复杂的解释？

**B3. 逻辑流程**

- 信息是否按逻辑顺序呈现？
- 相关主题是否分组在一起？
- 是否最小化了前向引用（引用后面才解释的内容）？

**B4. 示例与图解**

- 复杂概念是否具有说明性示例？
- 示例是否正确并与描述的行为一致？
- 在有帮助的地方是否提供了边缘情况示例？

**B5. 术语一致性**

- 术语在整篇文档中是否一致使用？
- 术语是否与代码库中使用的术语匹配（类名、函数名、枚举值）？
- 缩略词是否在首次使用时定义？

---

## 优先级 C：格式与风格

润色和文体一致性。对内容质量的影响较低。

**C1. 格式一致性**

- 标题、列表和表格格式是否一致？
- 代码片段是否正确格式化并进行了语法高亮？

**C2. 语法与措辞**

- 是否存在语法错误或不恰当的措辞？
- 写作风格是否始终一致？
- 语气是否适合目标受众？

**C3. 章节组织**

- 章节大小是否适当（不太长也不太短）？
- 目录（如果存在）是否与实际章节一致？
- 已弃用或过时的章节是否已清理？

---

## 排除清单

> **重要**：在上下文中加载的项目规则优先于此排除清单。

**默认排除的问题类型：**

1. 不影响可读性的纯格式偏好
2. 不提高清晰度或准确性的文体重写
3. 基于假设的未来需求而非当前代码的建议
4. 添加超出文档声明范围内容的建议

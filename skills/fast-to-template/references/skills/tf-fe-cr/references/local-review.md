# 本地代码评审

单代理本地变更评审。审查差异，展示确认的问题，并让用户交互式选择需要修复的问题。

## 参考资料

| 文件                 | 用途                                   |
| -------------------- | -------------------------------------- |
| `code-checklist.md`  | 代码评审检查清单                       |
| `doc-checklist.md`   | 文档评审检查清单                       |
| `judgment-matrix.md` | 风险等级、是否值得修复的标准、特殊规则 |

---

## 第一步：确定评审范围

根据 `$ARGUMENTS` 和工作树状态确定要评审的差异：

- **`$ARGUMENTS` 为空**，**存在未提交的变更**：评审范围仅为未提交的变更。使用 `git diff HEAD` 获取（已暂存 + 未暂存的已跟踪文件）。同时使用 `git status --porcelain` 检查未跟踪文件（`??` 行）并读取其内容进行评审。
- **`$ARGUMENTS` 为空**，**无未提交的变更**：按顺序检查常见基础分支来查找基础分支：`main`、`master`。使用第一个存在的分支。获取分支差异：
  ```
  git merge-base origin/{base_branch} HEAD
  git diff <merge-base-sha>
  ```
  同时使用 `git status --porcelain` 检查未跟踪文件（`??` 行）。
- **提交哈希**（如 `abc123`）：使用 `git rev-parse --verify` 验证，然后执行 `git show`。
- **提交范围**（如 `abc123..def456` 或 `abc123...def456`）：验证两个端点。获取包含两个端点的差异：
  ```
  git diff A~1..B
  ```
- **文件/目录路径**：验证所有路径在磁盘上存在，然后读取文件内容。

如果差异为空 → 显示使用示例并退出：
`/cr`（未提交的变更或当前分支）、
`/cr a1b2c3d`、`/cr a1b2c3d..e4f5g6h`、
`/cr src/foo.cpp`、`/cr 123`、`/cr https://github.com/.../pull/123`。

---

## 第二步：评审

评审差异。对代码文件应用 `code-checklist.md`，对文档文件应用 `doc-checklist.md`。当变更行依赖周围上下文时，根据需要阅读相关章节或相关定义。未跟踪文件没有差异——将其完整内容作为新代码进行评审。

对于发现的每个问题：

- 从当前树提供代码引用（文件:行号 + 代码片段）。
- 通过重新阅读代码进行自我验证——确认或撤销问题。
- 如果引用的路径/行号不再存在，在报告前通过 `git diff --name-only` 或文件搜索定位正确的文件/路径。

**输出规则**：仅向用户展示最终确认的问题。不要输出分析过程、排除推理或已考虑但被排除的问题。

---

## 第三步：过滤

查阅 `judgment-matrix.md` 进行风险评估、是否值得修复的标准判断以及特殊规则应用。丢弃不值得报告的问题。

如果过滤后没有剩余问题 → 报告"未发现问题"并退出。

---

## 第四步：报告与修复

展示评审内容摘要，然后列出所有确认的问题。使用**单个多选问题**询问需要修复的问题，其中每个选项的标签为问题摘要（如 `[风险] 文件:行号 — 描述`）。用户在一个提示中勾选多个选项。

如果用户选择了任何问题，应用相应的修复。

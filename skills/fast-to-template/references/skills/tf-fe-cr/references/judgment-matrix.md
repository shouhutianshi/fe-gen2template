# 判定矩阵

## 风险等级评估

| 风险 | 判定规则                                     | 示例                                                                                                                                                               |
| ---- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 低   | 仅存在一种合理的修复方式                     | 空值检查、修复错误注释、重命名以符合规范、移除冗余重复代码、添加 `reserve`、修复明显的差一错误                                                                     |
| 中   | 存在多种修复方式，但不涉及设计决策或外部契约 | 提取跨函数共享逻辑、移除未使用的内部方法、简化跨函数控制流、调整内部模块边界                                                                                       |
| 高   | 涉及设计决策或外部契约                       | 公共 API 变更（签名、行为、弃用）、测试基线变更、架构重构、存在多种可行方案的算法替换、引入新依赖、修改数据持久化/序列化格式、变更线程模型、涉及时空权衡的性能优化 |

## 按风险等级的处理方式

| `FIX_MODE` | 低风险   | 中风险   | 高风险   |
| ---------- | -------- | -------- | -------- |
| full       | 自动修复 | 自动修复 | 自动修复 |
| low_medium | 自动修复 | 自动修复 | 需确认   |
| low        | 自动修复 | 需确认   | 需确认   |

**"full" 模式特殊规则**：凡涉及测试基线变更（截图对比、黄金文件）的问题，无论风险等级如何，均需用户确认。

## 是否值得修复？

Code-checklist 和 doc-checklist 定义了**查找什么**。本节定义**是否修复**发现的问题。风险等级按问题划分，而非按类型（"逻辑错误"可能是低风险也可能是高风险）。

### 决策原则

1. **必须修复** — 问题影响运行时正确性、安全性或安保性。对应清单优先级 A 项。
2. **明确时修复** — 问题提升代码质量（性能、简化、架构）。仅在解决方案明确且不引入新风险时修复。性能变更需要对语义等价性有高度把握。对应清单优先级 B 项。
3. **不一致时修复** — 问题涉及命名、初始化、注释或文件组织。仅在违反上下文中加载的项目规则或与周边代码既定模式冲突时修复。对应清单优先级 C 项。
4. **始终跳过** — 纯粹的风格偏好（不违反任何一致性规则）、基于假设的未来需求而非当前代码的建议、以及无正确性问题的稳定代码的替代实现重写。

### 例外情况

- 重复代码提取：当相同逻辑明显重复时修复（不按数量阈值判定——按复杂度和维护成本判定）。
- 非错误修复的公共 API 签名变更：仅在能给 API 使用者带来明确收益时修复。始终为高风险。
- 测试覆盖缺口和回归风险需**标记，不修复**——向用户报告以供知晓，而非自动修复。

## 特殊规则

- 跨模块重命名：作为原子任务分配给单个修复者
- 之前轮次回滚的问题：**本轮不再尝试**

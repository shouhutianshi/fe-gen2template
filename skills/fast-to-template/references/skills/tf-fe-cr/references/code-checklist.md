# Code Review 检查清单

Reviewers 需检查本清单中的所有项目。A/B/C 级别表示审查优先级 —— 从 A 级（影响最大）开始，然后 B 级，最后 C 级。Reviewer 提示词会指定需要检查的级别。

测试代码的审查要求可适当放宽 —— 仅关注明显的实现错误。

**语言适用性**：标有语言标签（如 `[C/C++]`）的项目仅适用于该语言。未标记的项目适用于所有语言。跳过与项目语言不匹配的项目。

**项目特定规则**：已加载到 agent 上下文中的项目规则优先于此通用清单。当项目规则对特定问题类型有明确要求时，应遵循项目规则。

---

## 优先级 A：正确性与安全性

直接影响运行时行为的问题。影响最高 —— 首先审查这些。

**A1. 代码正确性**

- 函数返回值 / 输出参数是否在所有分支（包括错误分支）中正确设置？
- 实现是否与函数名 / 注释描述的行为一致？
- 条件逻辑表达式是否正确（&& / || 混淆、缺少否定、运算符优先级错误）？
- switch/case 语句是否覆盖所有分支？是否存在非预期的 fall-through？

**A2. 边界条件**

- 除零保护（浮点数和整数）
- 空容器操作（在 front() / back() / operator[] 前检查 empty()）`[C/C++]`
- Null/nil/undefined 解引用
- 整数溢出 / 下溢（尤其无符号减法 `[C/C++]`）
- 数组 / 字符串越界访问

**A3. 错误处理**

- I/O 操作后是否检查错误状态？
- 字符串 / 数据解析后是否验证结果有效性？
- 是否验证外部输入的合法性？
- 函数调用失败时是否有合理的回退 / 返回值？
- promises / 异步调用是否正确 await 并进行错误处理？`[JS/TS/Python/async languages]`

**A4. 注入与敏感数据** `[Web]`

- 用户输入是否未经清理就插入 DOM（innerHTML、dangerouslySetInnerHTML、v-html、[innerHTML]、document.write 等）？
- URL 参数、localStorage 或 postMessage 数据是否未经验证就直接使用？
- 客户端代码中是否硬编码了 API 密钥、令牌或凭据？

**A5. 资源管理**

- 手动分配的资源是否在所有路径上正确释放（优先使用 RAII）？`[C/C++]`
- 文件句柄 / 系统资源是否正确关闭？
- 锁的获取和释放是否正确配对？`[C/C++/Java/Go]`
- 数据库连接 / 网络套接字是否在 finally/defer 块中正确释放？
- 事件监听器、定时器、订阅和观察者是否在组件卸载或作用域退出时清理？`[Web]`

**A6. 内存安全** `[C/C++]`

- Use-after-move：移动后的对象是否被再次使用？
- 悬空引用 / 指针：是否返回指向局部变量的引用 / 指针？
- 容器修改后，容器元素引用是否仍在使用？

**A7. 线程安全**（如适用）

- 共享数据是否受到并发访问保护？
- 可变共享状态是否存在竞态条件？

**A8. 公共 API 注释准确性**

- 公共 API 注释是否准确描述了当前行为、参数和返回值？
- 注释中的值范围、约束和错误条件是否与实际实现一致？
- 相应 API 行为变更时是否更新了注释？

## 优先级 B：重构与优化

代码质量、性能和可维护性的改进。影响中等。

**B1. 性能优化**

- 容器：大小可预测时是否预分配空间？
- 不必要的深拷贝：**报告前必须对语义等价性有高度把握**
- 循环内重复计算：表达式是否可以移到循环外？
- 字符串操作：循环内频繁拼接是否可以优化？
- 不必要的临时对象构造 `[C/C++]`
- Const 引用：大对象是否按值传递而应使用 const& 以提升性能？（const 正确性约定见 C7）`[C/C++]`
- 组件是否因缺少 memoization、不稳定引用或 props 中内联对象/函数创建而不必要地重新渲染？`[React/Vue/Web]`
- 大型依赖是否完整导入而仅使用其中一小部分（tree-shaking 低效）？`[Web]`

**B2. 代码简化**

- 重复代码：明显重复的相同逻辑应提取（根据复杂度和维护成本判断，而非数量阈值）
- 深层嵌套 if/else：可以用提前返回简化
- 冗余条件检查：可以合并或消除的逻辑分支
- 过长的函数：单一职责的逻辑块可以提取为子方法
- 可以合并的相似分支逻辑

**B3. 模块架构**

- 模块职责是否清晰？是否存在职责边界违反？
- 依赖方向是否合理？
- 是否存在循环依赖？

**B4. 接口使用**

- 调用的 API 是否按其设计意图和文档使用？
- 是否使用了已弃用的接口？

**B5. 接口变更**

- 公共 API 签名或类接口是否有变更？
- 如有，描述变更及其范围（协调员将评估是否合理）

**B6. 测试覆盖** _（仅标记 —— 报告以供知晓，不自动修复）_

- 变更的逻辑路径是否有相应测试用例？
- 边界条件是否有测试覆盖？
- 错误路径是否有测试覆盖？

**B7. 回归风险** _（仅标记 —— 报告以供知晓，不自动修复）_

- 修改是否可能影响其他调用者？
- 行为变更在所有目标平台上是否一致？

**B8. 渲染正确性** `[Web]`

- 列表项是否使用稳定、唯一的 key 渲染（非数组索引）？
- 副作用是否正确放置在生命周期 hooks / useEffect 中并具有适当的依赖数组？
- 组件状态是否正确派生（无过时闭包、无不同步的派生状态）？

## 优先级 C：约定与文档

编码标准和文档一致性。对功能影响较低。

**C1. 命名约定**

- 代码是否遵循项目现有的命名风格（如已加载的项目规则中所定义）？
- 变量名语义是否清晰，避免不必要的缩写？
- 新代码中的命名是否与同一文件中的风格一致？

**C2. 初始化约定**

- 变量在声明时是否赋初值（按项目规则要求）？
- 类成员变量是否在声明时或构造函数中初始化？`[C/C++/Java]`

**C3. 项目语言约定**

- 代码是否符合项目规则中定义的语言使用限制（因项目而异）？
- 新代码是否与项目中的现有模式一致？

**C4. 注释约定**

- 公共 API 是否有足够的参数和返回值描述？
- 仅凭代码不足之处是否有设计意图说明？

**C5. 文件组织**

- 实现文件中的函数顺序是否与声明顺序一致？`[C/C++]`
- 头文件是否有适当的 include 守卫？`[C/C++]`
- include / import 依赖是否必要且合理？

**C6. 类型安全**

- 隐式窄化转换（大类型 -> 小类型）`[C/C++/Java]`
- 有符号 / 无符号混合比较 `[C/C++]`
- 魔术数字：硬编码值应提取为命名常量（除非上下文已使其清晰）

**C7. Const 正确性** `[C/C++]`

- 不修改状态的方法是否标记为 const？
- 不修改的参数是否作为 const 引用传递？
- 不修改的局部变量是否声明为 const？

**C8. 文档一致性**

- 代码中的类型名 / 枚举名是否与项目文档一致？
- 注释中的值范围是否与规范文档一致？

**C9. 可访问性** `[Web]`

- 图片是否有有意义的 alt 文本（或装饰性图片使用空 alt）？
- 表单输入是否有关联的 label？
- 交互元素是否可通过键盘导航并使用语义化 HTML？

## 排除清单（适用于所有级别）

> **重要**：以下项目为默认排除项。如果上下文中加载的项目规则对特定问题类型有明确要求（例如"不允许 lambda"、"变量必须初始化"），则该问题类型**不被排除**，应按照项目规则进行审查。项目规则优先于此排除清单。

**默认排除的问题类型：**

1. 纯样式偏好（在格式化工具范围内，且项目规则未明确要求）
2. 项目格式化工具已处理的格式问题（缩进、空白、换行等）
3. 基于假设的未来需求而非当前代码的建议
4. 遵循项目现有风格但不匹配某些外部标准的代码
5. 测试代码中的优先级 C 问题（除非项目规则有特定要求）
6. 对现有稳定代码的"更好的替代实现"建议（如果当前实现无 bug）

# Frontend Architect

前端架构师，根据现有项目规范和产品需求，生成或更新《全局架构协议》作为后续拆分功能模块、编写具体技术方案的"母本"。

## 适用场景

- 新项目启动前的技术架构设计
- 已有项目接到增量需求，需要评估架构是否需要调整
- 多人协作项目需要统一架构约束
- 模块划分不清晰导致耦合严重
- 需要一份"母本"文档来指导后续技术方案

## 使用方式

### 输入

用户提供以下信息：

1. **项目规范文件**：现有技术栈说明、目录结构规范、代码风格约束
2. **产品需求文档**：PRD、Feature List 或用户故事
3. **（可选）架构约束**：团队规模、技术债务限制

### 输出

《全局架构协议》Markdown 文档，包含：
1. 项目技术底座（沿用与约束）
2. 核心业务模块划分表（包含输入/输出定义）
3. 全局通信与状态协议
4. 公共组件与 Hooks 预案
5. 变更记录

## 工作流程

### Step 1: 存量架构评估

在开始任何设计工作之前，先检查当前项目是否已存在架构设计文档：

1. **查找现有文档**：检查 `docs/architecture/` 目录下是否已有 `overview.md` 或其他架构协议文档
2. **判断需求类型**：
   - **无现有文档** → 判定为全新需求，进入 Step 2 从零开始设计
   - **有现有文档** → 读取现有架构协议，进入差异分析
3. **差异分析**（仅增量需求）：
   - 将新需求与现有架构协议逐项对比
   - 识别哪些模块受影响、是否需要新增模块、通信协议是否需要调整
   - 生成变更影响摘要
4. **与用户沟通澄清**：
   - 向用户展示差异分析结果
   - 确认以下关键问题：
     - 本次需求是否需要新增模块？
     - 现有模块的职责边界是否需要调整？
     - 全局状态/通信协议是否需要变更？
     - 公共 Hooks/组件是否需要新增或修改？
   - 等待用户确认后再进入下一步

### Step 2: 规范提取

分析现有技术栈、目录结构、公共工具和 Hooks：

| 分析维度 | 具体内容 |
|---------|---------|
| 技术栈 | 框架（Vue/React）、构建工具、状态管理、路由方案 |
| 目录结构 | 现有模块划分、文件组织规范 |
| 公共工具 | Utils、Helpers、Common Functions |
| Hooks/Composables | 现有可复用逻辑抽象 |
| 样式规范 | CSS 方案、Design System、主题变量 |

### Step 3: 模块聚合

将零散的 PRD 需求/页面，按业务逻辑聚合成 **3-5 个核心大模块**：

```
聚合原则：
- 业务边界清晰，独立团队可并行开发
- 模块间依赖单向，不产生循环依赖
- 高内聚：同一模块的功能在同一目录下
- 低耦合：模块间通过标准接口通信
```

增量需求场景下，优先将新功能归入现有模块，仅在职责无法匹配时新增模块。

### Step 4: 协议制定

#### 4.1 全局数据流定义

| 存储位置 | 适用场景 | 示例 |
|---------|---------|------|
| 全局 Store (Pinia/Redux) | 跨页面共享状态、用户态、权限信息 | 用户信息、登录态、主题配置 |
| 模块级 Store | 同模块内多页面共享 | 购物车、当前选中课程 |
| 组件内 State | 纯 UI 状态、不需要共享 | 展开/收起、临时输入 |
| URL Params | 页面跳转传参、书签场景 | listId=123、tab=detail |
| Context/Provide | 跨层级传值、无需全局状态 | 某个弹窗的配置 |

#### 4.2 通信契约

定义模块间跳转的传参标准：

| 跳转场景 | 传参方式 | 参数标准 |
|---------|---------|---------|
| 页面跳转 | URL Query / Params | 遵循 RESTful 规范 |
| 模块内跳转 | Store 共享 | 通过 getter/action |
| 跨模块跳转 | 消息总线 / Event | 解耦调用 |
| 需要持久化 | LocalStorage / Cookie | 敏感信息加密 |

#### 4.3 公共能力预设

根据需求提前声明需要封装或复用的公共 Hooks/Utils：

| Hook/Utils 名称 | 职责 | 输入 | 输出 |
|----------------|------|------|------|
| useAuth | 鉴权与权限 | - | { isLogin, userInfo, hasPermission } |
| useFetch | 统一请求封装 | url, params | { data, loading, error, refetch } |
| usePagination | 分页逻辑 | total, pageSize | { page, setPage, reset } |
| useModal | 弹窗管理 | - | { open, close, setContent } |
| useToast | 轻提示 | message, type | { show, hide } |

## 注意事项

- **尊重存量代码**：协议必须基于现有项目规范，不是推翻重来
- **增量开发优先**：新规范作为增量，不修改已稳定运行的存量代码
- **先评估后动手**：每次收到新需求，必须先完成 Step 1 的存量评估和用户沟通，再开始设计
- **模块数控制**：建议 3-5 个核心大模块，避免过度设计
- **变更可追溯**：每次架构变更必须在变更记录中留痕，方便团队回溯决策

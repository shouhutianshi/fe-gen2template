# 前端技术方案文档生成器

将 PRD、Story、Bug 或需求描述转化为可指导开发、联调、验证和审查的前端技术方案。只设计前端；后端只写前端依赖的接口契约和行为结论。

## 快速开始

1. **生成模板**：`bash scripts/generate-template.sh docs/prd/<name>-tech-spec.md <项目名>`
2. **基线扫描**：`bash scripts/baseline-scan.sh` → 结果填入模板「现状基线」
3. **填写方案**：按模板章节逐步展开，每步读对应的约定文档（见 [references/conventions-guide.md](references/conventions-guide.md)）
4. **自检**：按 [references/self-check.md](references/self-check.md) 逐项检查
5. **审查**：`/cr` 审查方案

复杂需求（多模块、多人协作）按 [references/workflow.md](references/workflow.md) 走完整门禁流程。

## 适用边界

**用**：技术方案 / tech spec / PRD 转前端设计 / 重构方案 / 性能优化方案 / "帮我设计一下这个需求"。

**不用**：只拆模块边界 → `kit-fe-prd-split`；只看架构 → `kit-fe-arc`；写代码/修 bug → 开发流程。

## 核心原则

- **事实优先**：用户指令 > PRD > 架构文档 > 现有代码 > YApi > 标注假设。冲突时列出来，不自己合并——模型不知道的事实写成"待确认项"比编造好一百倍。
- **契约唯一**：同一概念只有一套枚举、字段名、接口路径。功能拆解、数据模型、状态流转之间不能打架。
- **现有优先**：目录结构、组件库、请求封装、测试命令以仓库实际代码为准，不假设理想状态。
- **约定优先**：先读 `CLAUDE.md`、`memory/MEMORY.md`、[references/conventions/](references/conventions/)。项目硬规则与通用默认冲突时按硬规则走。
- **范围不扩大**：只解决明确列出的问题。范围外内容进附录，不默认交付。

## 输入准入

生成前收集输入来源。缺失时写待确认项，不编造。

- **PRD**：有帮帮文档链接用 `kit-zyb-docs` 导出；无 PRD 基于用户描述生成草案并标注不完整。
- **Story/Bug**：用户提供 ID；查关联 Bug 用 `kit-zyb-pms`。
- **架构**：读 `docs/architecture/overview.md`；缺失用 `kit-fe-arc` 补或标记缺口。
- **模块边界**：复杂需求读 `docs/architecture/boundaries.md`；缺失用 `kit-fe-prd-split`。
- **接口**：YApi MCP 或现有代码。未定时写待确认项。
- **约定**：`CLAUDE.md` + `memory/MEMORY.md` + `conventions/*`。
- **基线**：执行 `bash scripts/baseline-scan.sh`，结果写入方案「现状基线」。

## 生成流程

| 阶段 | 做什么 | 什么时候算完成 |
| --- | --- | --- |
| 1. 输入盘点 | 收集上述输入 + 执行基线扫描 | 每个关键事实有来源或标注为假设 |
| 2. 范围确认 | 明确本期/后续、不做范围、端类型 | 3-6 条可验证的成功标准 |
| 3. 契约冻结 | 对齐路由、接口、枚举、权限 key、状态机 | 同一概念只有一处定义 |
| 4. 方案展开 | 按模板逐章生成 | 每个改动项有实现 + 影响面 + 验证 + 回滚 |
| 5. 验证设计 | 发现项目脚本，制定验证矩阵 | 命令来源明确 |
| 6. 自检审查 | 按 self-check.md 检查 + `/cr` 审查 | Critical 问题关闭 |

默认输出到对话。用户要求落盘时写入 `docs/prd/<story-or-kebab-name>-tech-spec.md`。

## 方案章节骨架

`〇` 背景目标边界 → `一` 页面总览（含页面还原三表）→ `二` 全局工程 → `二·甲` 工程基线 → `二·乙` 性能与稳定性 → `三` 功能拆解（每个改动项覆盖十个维度：实现/API/响应/逻辑/交互/验收/边界/影响面/回滚/测试）→ `四` 数据模型 → `五` 状态流转 → `六` 一致性索引 → 附录。

完整模板通过脚本生成，不适用章节保留标题写"本章不涉及"。

## 待确认项

遇到以下情况写入待确认项表而非编造：PRD 缺字段/权限/错误态、接口路径/枚举/错误码未定、共享契约 owner 未定、兼容/灰度/埋点/回滚策略未明确。

影响接口、状态机、权限、路由的未知项是阻断项——不关闭时方案不能进入开发。

## 前后端边界

写：组件/路由/状态/交互/样式/API 调用/错误处理/测试。不写：数据库/后端算法/中间件/CI 流水线。幂等/原子性只写前端依赖的结论。

## 与工作流对齐

复杂 PRD 先补架构和模块地图再进入方案。多 agent 并行需文件所有权不重叠 + 共享契约冻结。验证命令从 `package.json` 或 CI 发现。完成声明必须基于新鲜验证证据。提交前 `/cr` 审查。

## 版本管理

本 skill 使用 semver 版本号（frontmatter `version` 字段）。任何修改都应同步更新版本号：
- minor：新增章节、参考文档、脚本
- patch：措辞修正、补全、小幅优化
- major：删除章节、破坏性变更

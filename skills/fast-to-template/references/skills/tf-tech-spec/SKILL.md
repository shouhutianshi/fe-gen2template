# 前端技术方案文档生成器

根据 PRD、Story、Bug 或需求描述，生成可指导前端开发、联调、验证和审查的技术方案。该 skill 只设计前端方案；后端内容只写前端依赖的接口契约和行为结论。

完整文档模板见 [references/tech-spec-template.md](references/tech-spec-template.md)。工作流依赖见 [references/workflow.md](references/workflow.md)，生成方案时直接按该文件执行。

## 适用边界

使用本 skill：
- 用户要求"技术方案"、"技术文档"、"tech spec"、"技术设计"、"方案文档"、"实现方案"。
- 用户提供 PRD / Story / Bug / 需求描述，要求拆成前端可实现方案。
- 用户要从现有业务文档、YApi 或代码结构推导前端页面、组件、接口、状态、测试和上线方案。

不使用本 skill：
- 只需要拆 PRD 模块边界时，优先使用 `kit-fe-prd-split`。
- 只需要全局架构概览时，优先使用 `kit-fe-arc`。
- 只需要写具体代码、修 bug 或做代码审查时，使用对应开发、调试或 `cr`（命令 `/cr`）流程。

## 核心原则

- **事实优先**：用户指令 > PRD/Story/Bug > 架构文档 > 现有代码 > YApi/接口文档 > 明确标注的假设。来源冲突时必须列出冲突，不能自行合并。
- **未知可见**：接口、字段、权限、埋点、兼容范围、上线策略等未确定时，写入"待确认项"，不要编造。
- **契约唯一**：同一业务概念只能有一套枚举值、字段名、接口路径和权限 key。汇总表、功能章节、数据模型必须一致。
- **现有优先**：目录、组件库、状态管理、请求封装、构建脚本和测试命令必须以当前仓库实际代码、`package.json` 和 CI 配置为准。
- **前端边界**：写组件、路由、状态、交互、样式、接口调用、错误处理、测试和验证；不写数据库、后端算法、服务端中间件或后端实现细节。
- **可审计**：方案必须能追踪到需求来源、接口来源、验证命令和风险 owner。

## 输入准入

开始生成前，先收集并记录输入来源：

| 输入 | 获取方式 | 缺失处理 |
| --- | --- | --- |
| PRD / 帮帮文档 | 有 docs.zuoyebang.cc 链接时使用 `kit-zyb-docs` 导出并记录 URL、fileId、导出时间 | 无 PRD 时只基于用户描述生成草案，并标记为"需求来源不完整" |
| Story / Bug | 用户提供 ID 或上下文；需要查关联 Bug 时使用 `kit-zyb-pms` | 无 ID 时记录为人工输入 |
| 架构概览 | 读取 `docs/architecture/overview.md`；不存在时先用 `kit-fe-arc` 生成或标记缺口 | 不允许跳过架构差异影响 |
| 模块边界 | 复杂需求读取 `docs/architecture/boundaries.md`；缺失时先用 `kit-fe-prd-split` 生成模块地图 | 共享契约未稳定前不能给出并行实现计划 |
| 接口契约 | 使用 YApi MCP 或现有接口文档/代码 | 接口未定时写待确认项，禁止虚构路径和字段 |
| 代码现状 | 读取 README、`docs/conventions/*`、`package.json`、相关 `src/**` | 不能读取时标记风险和影响 |

复杂需求按本 skill 的 `references/workflow.md` 走完整路径：PRD 读取 -> 需求拆解 -> 技术方案 -> 实施计划 -> 开发验证 -> 联调审查。单模块小需求可走快速路径，但仍必须保留输入来源、待确认项和验证矩阵。

## 生成流程

| 阶段 | 动作 | 产物 | 退出条件 | 失败处理 |
| --- | --- | --- | --- | --- |
| 1. 输入盘点 | 读取 PRD、Story/Bug、架构、模块边界、YApi、现有代码和约定文档 | 输入来源表 | 每个关键事实有来源或被标记为假设 | 缺权限时停下说明；缺材料时列待确认项 |
| 2. 范围确认 | 明确本期/后续、不做范围、端类型、技术栈、影响模块 | 目标边界与成功标准 | 范围能被 3-6 条可验证标准覆盖 | 范围冲突时先问用户，不能继续细化 |
| 3. 契约冻结 | 对齐路由、接口、枚举、数据模型、权限 key、埋点口径、状态机 | 主契约表和冲突清单 | 同一概念只有一处定义 | YApi/PRD/代码冲突时列冲突和推荐处理 |
| 4. 方案展开 | 按模板生成页面、工程、功能、数据、状态、测试、上线章节 | 技术方案正文 | 每个功能点包含实现、API、响应、逻辑、交互、验收、边界 | 信息不足时写待确认项，不补虚假细节 |
| 5. 验证设计 | 发现项目脚本，制定 lint/typecheck/test/build/browser smoke 矩阵 | 验证矩阵 | 命令来源明确，未验证项有风险说明 | 没有脚本时记录"未发现脚本"，不臆造命令 |
| 6. 自检审查 | 跑方案自检清单，必要时使用 `cr`（命令 `/cr`）做文档/方案审查 | 自检结果和剩余风险 | Critical/Important 问题关闭或有技术反驳 | 有阻断风险时不能声明方案可执行 |

默认输出到对话；如果用户要求落盘或上下文已有需求文档目录，写入 `docs/prd/<story-or-kebab-name>-tech-spec.md`。文件名优先使用 Story/Bug ID，其次使用需求英文 kebab-case 摘要。

## 未知与冲突处理

遇到以下情况必须显式处理：

- PRD 缺页面、字段、权限、错误态、验收标准。
- PRD 与现有代码、架构文档、YApi 或用户最新指令冲突。
- 接口路径、请求参数、响应字段、错误码、枚举值未定。
- 共享契约涉及多个模块，但模块边界或 owner 未定。
- 基线验证失败、历史债务会影响本次方案。
- 兼容范围、i18n、灰度、埋点、回滚、安全策略未明确。

使用待确认项表：

| ID | 问题 | 来源/冲突 | 影响 | 推荐处理 | Owner | 状态 |
| --- | --- | --- | --- | --- | --- | --- |

规则：
- 不确定事实不得写成确定设计。
- 可以提出推荐方案，但必须标注"推荐"和依据。
- 影响接口、状态机、权限、路由、上线策略的未知项属于阻断项；先问用户或标记方案不可进入开发。
- 本期范围外可以进入附录，但不得在功能拆解中默认设计成必须交付。
- 不泄露 Cookie、Token、账号、内部链接敏感内容或未脱敏截图；引用内部文档时只保留必要摘要和来源标识。

## 技术方案必含内容

生成文档时使用中文编号作为顶级章节，不适用章节保留标题并写"本章不涉及"：

- **〇、背景、目标、边界与验收口径**：需求来源、现状问题、目标/非目标、端范围、技术栈、成功标准、术语、假设和待确认项。
- **一、页面总览**：路由、入口、布局、全局鉴权、loading、异常、状态保留、兼容范围。
- **二、全局工程方案**：应用边界、目录建议、模块边界、文件所有权、路由跳转、接口联调约束、权限矩阵、安全、性能、埋点、部署灰度回滚、测试策略、风险清单。
- **三、功能拆解**：每个模块/页面逐项描述实现、API、响应、逻辑、交互、验收、边界和测试所有权。
- **四、核心数据模型**：ID、时间、金额、枚举、TypeScript interface、实体关系和来源。
- **五、前端状态流转**：状态机、转移触发点、并发冲突、`409` 处理、权限约束、API 表。
- **六、功能一致性索引**：PRD -> 模块 -> 路由 -> 接口 -> 数据模型 -> 用例 -> 验收标准的追踪矩阵。
- **附录**：后续迭代、待确认项、术语、风险、参考链接。附录内容默认不进入本期开发。

每个功能点必须包含：

| 维度 | 要求 |
| --- | --- |
| 实现 | 组件名、关键 props、布局、样式 class 或既有组件封装 |
| API | 方法、路径、参数、请求头、幂等/版本参数 |
| 响应 | 前端期望的数据结构、错误结构、空数据结构 |
| 逻辑 | 排序、筛选、状态映射、权限判断、副作用 |
| 交互 | 点击、跳转、确认、hover、快捷键、移动端手势 |
| 验收 | 用户可见结果和可执行检查方式 |
| 边界 | 空数据、加载失败、超时、权限不足、并发冲突、兼容降级 |
| 测试 | unit/component/integration/e2e/manual 层级与 owner |

## 与工作流对齐

方案中必须体现本 skill `references/workflow.md` 的以下要求：

- 复杂 PRD 必须先有架构概览和模块地图；缺失时先补，不直接进入实现方案。
- 多 agent 并行只允许在文件所有权不重叠、共享契约冻结后进行。
- 技术方案需要给出文件所有权矩阵：模块、owner、允许修改、禁止修改、集成顺序、验证命令。
- 前后端联调必须列出 YApi/接口来源、请求/响应/错误码一致性检查项。
- 本地验证命令必须从 `package.json`、CI 或 `docs/conventions/verification.md` 发现；TypeScript 检查不要裸跑 `tsc`。
- 完成声明前只能基于新鲜验证证据下结论；只设计了验证矩阵时，不能说功能已验证。
- 提交前需要 `cr`（命令 `/cr`）审查；方案应留下审查入口和风险等级。

文件所有权矩阵格式：

| 模块 | Owner | 允许修改 | 禁止修改 | 共享契约 | 集成顺序 | 验证命令 |
| --- | --- | --- | --- | --- | --- | --- |

验证矩阵格式：

| 验证项 | 命令来源 | 推荐命令 | 通过标准 | 未执行风险 |
| --- | --- | --- | --- | --- |

## 自检清单

交付方案前逐项检查：

- PRD/Story/Bug 每个本期需求都能追踪到模块、路由/入口、接口、用例和验收标准。
- 主接口索引、功能拆解、数据模型、状态流转中的字段、枚举、路径无冲突。
- 权限矩阵覆盖路由、按钮、接口错误态和无权限降级表现。
- 每个写操作都有前端校验、版本/并发处理、成功反馈、失败反馈和重复提交保护。
- 每个列表都有 loading、empty、error、分页、筛选重置、请求取消和 URL 同步策略。
- 埋点、错误上报、性能、兼容、i18n、灰度、回滚未涉及时明确写"本章不涉及"。
- 验证矩阵命令来自实际项目脚本或约定文档；不存在脚本时已记录风险。
- 待确认项都有 owner、影响和推荐处理；阻断项未关闭时不声明方案可进入开发。
- 未泄露凭据、Cookie、Token、未脱敏内部链接或敏感截图内容。

## 前后端边界速查

| 内容 | 写不写 | 放哪 |
| --- | --- | --- |
| 组件选型、props、样式、布局 | 写 | 功能拆解 |
| 路由表、跳转参数、URL 状态 | 写 | 页面总览 / 全局工程 |
| 前端状态管理、状态机、并发处理 | 写 | 状态流转 |
| 前端格式化、校验、转换逻辑 | 写 | 功能拆解 / shared |
| API 路径、方法、请求参数、响应结构 | 写 | 联调约束 / 功能拆解 |
| HTTP 错误码前端处理方式 | 写 | 联调约束 |
| 权限 key、按钮显隐、无权限降级 | 写 | 权限矩阵 / 功能拆解 |
| 后端如何校验参数、实现接口 | 不写 | - |
| 数据库表结构、索引 | 不写 | - |
| 后端中间件、缓存策略 | 不写 | - |
| 后端如何保证幂等/原子性 | 只写前端依赖的结论 | 联调约束 / 事务边界 |
| CI/CD 流水线实现 | 只写部署入口、灰度和回滚结论 | 部署章节 |
